Step 1 – Dummy Tasks (individual tasks)

Goal: understand what a basic Celery task is.

  1. Click ▶ Run Dummy A on the dashboard.
  2. A toast notification appears bottom-right with the task ID.
  3. Go to Flower → Tasks tab → see tasks.dummy_a with status SUCCESS.
  4. Click the task ID in Flower → see: args, kwargs, result (10), runtime.
  5. Repeat for Dummy B (result: 6) and Dummy C (result: "Final result: 1").
  6. Scroll down → Audit Trail automatically shows the 3 records (auto-refreshes every 5 s).
💡 Every task writes a record via Celery signals (task_prerun / task_postrun) to the central AuditTrail — without the task explicitly doing so itself.

Step 2 – Task Chaining (A → B → C)

Goal: pass the result of one task as input to the next.

  1. Click ▶ Run Chain.
  2. In Flower → Tasks: you see 3 sequential tasks appear:
    • dummy_a(1) → result 10
    • dummy_b(10) → result 15 (10 + 5)
    • dummy_c(15) → result "Final result: 15"
  3. Notice each task only starts after the previous one finishes (sequential).
  4. Check the Audit Trail: 3 new rows with ascending timestamps.
💡 Canvas: chain(dummy_a.s(1), dummy_b.s(), dummy_c.s())

Step 3 – Group (parallel)

Goal: run multiple tasks simultaneously.

  1. Click ▶ Run Group.
  2. In Flower → Tasks: the 3 tasks (dummy_a, dummy_b, dummy_c) appear almost simultaneously and run in parallel.
  3. Compare start times in Flower — they overlap.
  4. Note the difference with Step 2: here there is no order and no result passing.
💡 Canvas: group(dummy_a.s(1), dummy_b.s(2), dummy_c.s(3))

Steps 4a / 4b / 4c – Chords

Step 4a – Chord (basic)

Goal: run tasks in parallel and then merge all results in a single callback.

  1. Click ▶ Run Chord.
  2. In Flower: see 3 parallel tasks + then tasks.chord_callback.
  3. Click on chord_callback → result: "Chord complete! Got 3 results: [20, 8, 'Final result: 4']"
  4. The callback waits until all header tasks are complete before starting.
💡 Canvas: chord(group(A, B, C), chord_callback.s())

Step 4b – Chord with chained header

Goal: each header task is itself a mini-chain.

  1. Click ▶ Run Chained Chord.
  2. In Flower: you see 3 pairs (dummy_a → dummy_b), then the callback.
  3. The callback sums the final values: Chain 1: 15, Chain 2: 25, Chain 3: 35 → Sum = 75
💡 Canvas: chord(group(chain(A,B), chain(A,B), chain(A,B)), callback.s())

Step 4c – Incomplete Chord

Goal: see what happens when a chord header does not fully complete.

  1. Click ▶ Send Incomplete Chord.
  2. In Flower → Tasks: you see always_failing_task with status FAILURE.
  3. The incomplete_chord_callback never appears — the chord is stuck.
  4. The chord appears in the "Manage Incomplete Chords" table on the dashboard.
⚠️ This is a deliberately created situation to demonstrate the management functions (Step 5).

Step 5 – Managing an Incomplete Chord

Goal: be able to clean up or force-complete a stuck chord.

Option A – Delete (on_failure handling)

  1. In the "Manage Incomplete Chords" table → click 🗑 Delete next to the chord.
  2. This calls force_delete_incomplete_chord: all header tasks are revoked, the chord is marked as resolved.
  3. In Flower → the chord disappears from active tasks.
  4. Audit Trail shows a SUCCESS record for tasks.force_delete_incomplete_chord.

Option B – Force Start

  1. First send a new incomplete chord (repeat Step 4c).
  2. Click ⚡ Force Start → this dispatches a new replacement chord so the callback finally executes.
  3. In Flower: see the new chord and its chord_callback with SUCCESS.

Step 6 – Tasks that spawn new tasks

Goal: show that a task can spawn new tasks at runtime (one-shot only).

Spawning Task

  1. Click ▶ Run Spawning.
  2. In Flower: spawning_task starts → then spawns 3 × dummy_a as children.
  3. A total of 4 tasks appear (1 parent + 3 children).
  4. depth=0 ensures children do not spawn again (no infinite loop).

Dynamic Workflow Task

  1. Click ▶ Run Dynamic.
  2. The task builds a group at runtime based on [1,2,3,4,5].
  3. In Flower: 1 dispatcher task + 5 parallel dummy_b tasks.

Step 7 – Celery Workflow (full canvas combination)

Goal: combine chord + chain in a single pipeline.

  1. Click ▶ Run Workflow.
  2. In Flower: follow the sequence:
    • Step 1: dummy_a(5) and dummy_b(5) run in parallel
    • Step 2: chord_callback aggregates the results
    • Step 3: the callback result is piped as input into dummy_c
  3. This shows a complete canvas pipeline: chord(...) | dummy_c

Steps 8–12 – Monitoring demos

Step 8 – Timeout Demo

Goal: see SoftTimeLimitExceeded in action and graceful handling.

  1. Click ▶ Trigger Timeout.
  2. The task tries to sleep for 20 s, but soft_time_limit = 5 s.
  3. In Flower → wait ~5 s → status SUCCESS (exception is caught internally).
  4. Click the task → result: "SoftTimeLimitExceeded after 20s – handled gracefully"
  5. Audit Trail shows FAILURE (we register it as an error in the audit).
💡 This allows a task to clean up gracefully on timeout instead of being killed abruptly.

Step 9 – Retry Demo

Goal: see automatic retries in Flower.

  1. Click ▶ Trigger Retry.
  2. In Flower → Tasks: the task appears multiple times:
    • Attempt 0: FAILURE → retry scheduled
    • Attempt 1: FAILURE → retry scheduled
    • Attempt 2: SUCCESS
  3. Click one of the attempts → see retries: 2 in the task details.
  4. In Flower → Tasks tab → filter on tasks.retry_demo → see all attempts.

Step 10 – Scheduled Task (Heartbeat)

Goal: automatically recurring task via django-celery-beat.

  1. Wait up to 1 minute.
  2. In Flower → Tasks: tasks.scheduled_heartbeat appears automatically every minute.
  3. Go to Django Admin → Periodic Tasks → see "Celery – Scheduled Heartbeat". You can adjust the interval, pause it or trigger it manually here.
  4. Audit Trail: every heartbeat writes a record with a timestamp.

Step 12 – Flower Remote Control

Goal: manage tasks from Flower.

  1. Start a Timeout Demo (active for ~5 s).
  2. Immediately go to Flower → Tasks → click the active task.
  3. Click Revoke → the task is stopped.
  4. Back on the dashboard: Audit Trail shows REVOKED.

Auto-refresh in Flower: top-right → set refresh interval (e.g. 2 s). Flower → Workers: real-time worker load, active/reserved tasks.

Step 11 – Central Audit Trail

Goal: overview of all executed tasks.

  1. Scroll down on the dashboard → Audit Trail table.
  2. Each row = one task execution with: task name, status (badge), task ID, result, timestamps.
  3. The table auto-refreshes every 5 seconds (toggle on/off via checkbox).
  4. Click ↺ Refresh now for an immediate update.
  5. Go to Django Admin → Audit Trail for full search and filter functionality.
💡 Records are written via Celery signals (task_prerun / task_postrun) — no explicit logging needed in any task.
